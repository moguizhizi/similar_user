# AGENTS.md

本文件用于说明：代码代理或自动化助手在本仓库中工作时，应遵循哪些项目级协作原则。

## 适用范围

- 这是一个新项目，仓库结构、依赖、脚本和测试能力都应以当前仓库中的实际内容为准。
- 不要把其他项目的目录约定、技术栈假设、文件职责或运行方式直接套用到这里。
- 在没有实际代码、配置或文档支撑时，不要把推断写成事实。

## 基本工作原则

- 优先做小而明确的修改，避免无必要的大范围重构。
- 先阅读当前仓库已有内容，再决定实现方式，不要根据模板经验直接下结论。
- 尽量沿用现有代码风格、命名方式和目录组织；只有在确有必要时再新增结构。
- 非必要不要引入新的重量级依赖；如果确实新增依赖，应同步更新依赖声明，并在交付说明中写清楚原因。
- 如果仓库中已有中文注释、中文术语或中文说明，尽量延续原有表达习惯，不要为了统一而强行改写。

## 实现与修改约定

- 新增逻辑应放在与其职责最接近的位置，避免把不相关的内容堆到单个文件中。
- 优先选择清晰、直接、易维护的实现，避免过度抽象。
- 函数和模块职责尽量单一。
- 只在必要处添加简短注释，说明为什么这样做，而不是重复代码表面含义。
- 不要把临时实验性质或一次性调试逻辑直接混入正式代码，除非任务明确要求。

## 测试与验证约定

- 不要默认仓库已经具备完整测试、脚本或可运行环境，必须以实际检查结果为准。
- 优先运行与本次改动最相关的最小范围验证。
- 如果当前仓库测试覆盖不足，优先补充有针对性的最小测试，而不是搭建空泛的大型测试框架。
- 如果因为依赖缺失、环境不完整、测试数据缺失或脚本尚未完成而无法验证，需要在最终说明中明确写出。
- 只有实际运行并确认通过的检查，才能在交付时描述为“已验证”。

## 交付说明要求

完成工作后，建议明确说明：

- 改了什么
- 实际运行了哪些测试或检查
- 哪些地方尚未验证，以及未验证的原因

如果某些判断来自代码结构推断而非仓库中的明确实现，也要在说明中直说，不要把推断表述成既有事实。

## 提交协作约定

- 每次完成代码修改并实际运行相关测试通过后，自动给出建议的 commit message，并询问用户是否提交。
- 只有用户明确确认后，才执行 `git add` 和 `git commit`。
- 提交成功后，需要推送到用户指定的远端分支；如果用户没有指定远端分支，先询问目标分支再执行 `git push`。
- 用户明确确认提交或推送后，直接执行对应的 `git add`、`git commit`、`git push`，不要在对话层面重复询问确认；如果用户未指定远端分支，默认推送到 `sk`。
- 提交时只暂存本次任务相关文件，不要包含无关的已有变更或未跟踪文件。
